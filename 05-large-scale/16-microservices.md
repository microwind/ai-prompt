# 16 微服务架构

> 高级 ⭐⭐⭐⭐⭐ | 适合大型系统拆分和服务治理

## 核心概念

微服务架构设计涉及多个维度的考虑：
- **服务边界**: 如何划分服务，保证内聚外疏
- **通信方式**: 同步(RPC/REST) vs 异步(消息队列)
- **数据一致性**: 分布式事务、最终一致性
- **服务治理**: 服务发现、负载均衡、容错降级
- **可观测性**: 链路追踪、日志聚合、监控告警

## 最佳实践

🎯 **原则1: 明确业务驱动**
- ❌ 差: "我想用微服务"
- ✅ 好: "公司有3个独立的产品线，每条线需要独立扩展和部署"

🎯 **原则2: 关键约束先行**
- ❌ 差: "设计电商系统"
- ✅ 好: "日均交易10万笔，支付交易强一致性要求，库存可最终一致"

🎯 **原则3: 说明现状和目标**
- ❌ 差: "我们要迁移到微服务"
- ✅ 好: "当前单体应用，部署周期2周，扩展能力受限。目标是周部署多次，核心服务独立扩展"

🎯 **原则4: 列举具体约束**
- ❌ 差: "设计一个可靠的系统"
- ✅ 好: "可用性要求99.9%，最多能接受15秒数据延迟，跨域部署"

## ✅ 优质Prompt示例

### 示例1: 服务拆分设计

**Prompt:**
```
我需要帮助设计一个中型SaaS产品的微服务架构。

业务背景：
- 项目管理系统（类似Jira）
- 当前单体架构，技术栈：Java + Spring Boot + PostgreSQL
- 业务模块：用户/权限、项目、任务、报表、通知、文件存储

现状问题：
1. 新功能部署需要整体测试，周期长（2周）
2. 任务创建高峰期（70%用户同时操作）导致整体应用卡顿
3. 报表查询特别复杂，影响其他功能的数据库性能
4. 不同部门的产品线需要独立定制，目前耦合度高

目标：
1. 支持周部署2-3次
2. 任务创建峰值处理1万QPS
3. 报表查询不影响核心业务
4. 支持A/B测试和灰度发布

约束条件：
- 现有开发团队20人，分为5个小组
- 不想大规模重写，希望循序渐进地迁移
- 数据一致性：订单数据强一致，活动数据可最终一致
- 预算有限，希望尽量使用开源方案

问题：
1. 应该如何拆分服务？
2. 每个服务的职责边界在哪里？
3. 迁移路径是什么？
4. 需要什么基础设施投入？
```

**预期输出:**
- 建议的服务拆分方案（包含服务列表和边界）
- 每个服务的核心功能
- 服务间的通信方式（同步/异步）
- 数据存储策略（每个服务独立DB还是共享）
- 迁移的阶段和优先级
- 所需的技术栈和工具

**适用场景:** 从单体向微服务迁移、大型系统重构、团队扩展

### 示例2: API网关设计

**Prompt:**
```
我们有15个后端微服务，需要设计一个API网关。

服务列表：
- auth-service: 认证与授权
- user-service: 用户管理
- order-service: 订单
- payment-service: 支付
- inventory-service: 库存
- notification-service: 通知
- search-service: 搜索
- analytics-service: 数据分析
- (其他7个...)

当前问题：
1. 移动端和Web端需要不同的API响应格式
2. 某些接口需要调用多个后端服务（用户信息+订单+推荐）
3. 认证检查重复实现在每个服务中
4. 监控和日志分散，难以追踪请求链路
5. 不同客户端（iOS/Android/Web）有不同的速率限制要求

要求：
1. 支持不同的API版本（v1/v2）
2. 实现BFF（Backend for Frontend）模式分离
3. 统一认证、授权、日志、监控
4. 支持请求合并和响应聚合
5. 实现熔断、限流、重试机制
6. 支持灰度发布和金丝雀部署

预期架构：
```
Client (Web/iOS/Android)
         ↓
    API Gateway
         ↓
  (BFF / Service Router)
         ↓
  [Auth] [User] [Order] ... [15个服务]
```

问题：
1. 应该用现成的网关（Kong/Nginx/Spring Cloud Gateway）还是自己开发？
2. BFF应该如何组织？
3. 如何实现跨服务的请求合并？
4. 认证token应该如何管理和分发？
5. 如何监控API网关本身的健康状态？
```

**预期输出:**
- 推荐的网关方案及理由
- API网关的核心功能设计
- BFF的组织方式和实现建议
- 认证授权流程设计
- 链路追踪的集成方式
- 部署和扩容策略

**适用场景:** 微服务通信管理、API聚合、多客户端支持

### 示例3: 分布式事务设计

**Prompt:**
```
我们面临分布式事务的问题，需要设计解决方案。

场景：用户下单，涉及多个服务
1. order-service: 创建订单
2. inventory-service: 扣减库存
3. payment-service: 支付扣款
4. notification-service: 发送确认邮件

当前实现的问题：
- 订单已创建但支付失败 → 数据不一致
- 库存扣减了但支付超时 → 库存为负
- 邮件发送失败 → 业务流程中断

要求：
1. 订单和支付必须强一致（钱和货对应）
2. 库存可以最终一致（允许短期超卖，之后补救）
3. 邮件可靠发送（失败自动重试，但不阻塞主流程）
4. 任何服务故障都不应导致系统不可用
5. 支持事务回滚和补偿

预期的交互流程：
```
User Submit Order
         ↓
[order-service] 创建订单 (状态: PENDING)
         ↓
[inventory-service] 预留库存 (状态: RESERVED)
         ↓
[payment-service] 扣款 (状态: PAID)
         ↓
[order-service] 确认订单 (状态: CONFIRMED)
         ↓
[notification-service] 异步发送邮件
```

问题：
1. 应该用什么模式？Saga模式vs TCC模式？
2. 如何实现补偿逻辑？
3. 消息丢失和重复消费如何处理？
4. 如何处理网络分区（某个服务长期不可用）？
```

**预期输出:**
- 推荐的一致性模式（Saga/TCC/Event Sourcing等）
- 详细的交互流程图
- 各种失败场景的处理方案
- 具体的实现建议（使用什么框架/工具）
- 幂等性和去重方案
- 监控和告警策略

**适用场景:** 订单系统、支付系统、库存系统、关键业务流程

### 示例4: 服务监控与告警

**Prompt:**
```
我们有20个微服务，需要设计完整的可观测性体系。

当前状况：
- 部分服务有应用日志，但没有统一收集
- 没有性能监控，故障时无法快速定位
- 没有链路追踪，跨服务请求难以debug
- 告警规则分散，经常误报或漏报

关键指标需求：
1. 延迟（Latency）: 99%请求应在100ms内返回
2. 吞吐（Throughput）: 支持1万QPS
3. 错误率（Error Rate）: 应维持在0.01%以下
4. 可用性（Availability）: 99.9%
5. 资源使用（CPU/Memory/Disk）

服务拓扑示意：
```
Gateway → [user-service] → user-db
       → [order-service] → order-db → [payment-service] → payment-api
       → [search-service] → elasticsearch
```

问题：
1. 应该如何收集和存储日志？（ELK Stack/Loki/CloudWatch）
2. 应该用什么监控系统？（Prometheus/Datadog/新遗迹）
3. 链路追踪如何实现？（Jaeger/Zipkin/Skywalking）
4. 告警规则如何设计才能避免误报？
5. 如何快速从告警定位到具体问题？
6. 成本如何控制（数据量很大）？
```

**预期输出:**
- 完整的可观测性架构设计
- 具体的技术选型及理由
- 各类指标的定义和采集方式
- 告警规则设计原则
- 仪表板和数据可视化建议
- 成本预估和优化建议
- 团队工作流程（故障排查流程）

**适用场景:** 生产环境监控、故障诊断、性能优化、容量规划

## ❌ 常见误区

### 误区1: 问题定义不清

❌ 不要：
```
"我需要微服务架构"
```

✅ 应该：
```
"我们现在是单体应用，部署周期长，某些模块经常是性能瓶颈，
需要能独立部署和扩展"
```

### 误区2: 忽视非功能需求

❌ 不要：
```
"设计服务拆分"
[完全没提性能、可靠性要求]
```

✅ 应该：
```
"服务拆分设计，要求：
- QPS: [数字]
- 延迟: [目标]
- 可用性: [要求]
- 数据一致性级别: [强/最终]"
```

### 误区3: 追求技术而忽视业务

❌ 不要：
```
"用Kafka做异步通信"
[不考虑业务需求]
```

✅ 应该：
```
"我们的业务需要[具体需求]，
为此需要异步通信，选择Kafka是因为..."
```

## 高级技巧

### 技巧1: 用架构决策记录(ADR)

```
针对[关键决策]，请帮我记录为ADR格式：

标题: [短标题]
背景: [为什么需要做决策]
选项:
  - 选项A: [优点] vs [缺点]
  - 选项B: ...
选中: [选项]
理由: [关键理由]
风险: [可能的负面影响]
后续: [验证决策的指标]
```

### 技巧2: 风险和失败模式分析

```
这个架构可能有什么问题？

请分析：
1. 单点故障：哪些点故障会导致整体不可用？
2. 性能瓶颈：什么情况下会成为瓶颈？
3. 成本问题：基础设施成本如何？
4. 运维难度：这个架构运维成本高吗？
5. 迁移风险：从当前架构迁移过来的风险？
```

### 技巧3: 具体的实施计划

```
我同意这个架构方案，现在需要制定实施计划。

约束：
- 团队规模：[人数]
- 时间框架：[期限]
- 不能停止当前业务开发

请给出：
1. 第一阶段做什么（Week 1-4）
2. 里程碑和关键指标
3. 所需的基础设施投入
4. 团队培训计划
5. 风险识别和应对
```

## 📌 微服务设计检查清单

设计微服务架构时，确认：
- [ ] 业务驱动清晰（为什么需要微服务）
- [ ] 关键约束明确（QPS、延迟、一致性等）
- [ ] 服务边界合理（高内聚、低耦合）
- [ ] 通信方式选择合理（同步/异步）
- [ ] 数据一致性有方案（分布式事务）
- [ ] 可观测性考虑周全（日志、监控、追踪）
- [ ] 故障恢复有预案（熔断、降级、重试）
- [ ] 部署流程明确（CI/CD、灰度发布）
- [ ] 团队组织适配（康威定律）
- [ ] 成本评估完整（基础设施+运维）
