# T03 诊断工具与技巧速查

## 🔧 编程语言诊断工具速查

### Java诊断工具集

**基础命令:**
```bash
# 查看Java进程
jps -l

# 查看JVM信息
java -version

# 获取堆转储
jmap -dump:live,format=b,file=heap.bin <pid>

# 查看线程信息
jstack <pid>

# 实时监控
jstat -gc -h10 <pid> 1000

# 查看JVM参数
jinfo <pid>
```

**分析堆快照:**
```bash
# 使用jhat分析
jhat heap.bin
# 浏览器访问 http://localhost:7000

# 或用Eclipse MAT等工具打开
```

**内存泄漏诊断:**
```
1. jmap导出堆快照
2. 用MAT/jhat分析大对象和引用
3. 查找保留集（retained set）
4. 在代码中定位对应的对象创建点
```

### Python诊断工具集

**调试工具:**
```python
import pdb
pdb.set_trace()  # 设置断点

# 或使用ipdb（更强大）
import ipdb
ipdb.set_trace()
```

**性能分析:**
```bash
# 查看函数调用耗时
python -m cProfile -s cumulative script.py

# 内存分析
python -m memory_profiler script.py

# 使用line_profiler逐行分析
kernprof -l -v script.py
```

**日志跟踪:**
```python
import logging
logging.basicConfig(level=logging.DEBUG)

# 或使用traceback输出详细堆栈
import traceback
try:
    ...
except Exception:
    traceback.print_exc()
```

### JavaScript/Node.js诊断

**内置调试:**
```bash
# 启用调试模式
node --inspect script.js
# 浏览器访问 chrome://inspect

# 或使用Node Inspector
node-inspect script.js
```

**性能分析:**
```javascript

```

**日志工具:**
```javascript

```

## 📊 诊断决策树

```
问题现象
├─ 程序崩溃（有异常）
│  ├─ NullPointerException → 检查对象初始化和判空
│  ├─ OutOfMemoryError → 分析堆快照，查找内存泄漏
│  ├─ StackOverflowError → 检查递归深度
│  ├─ DatabaseException → 检查连接、查询、锁
│  └─ 其他异常 → 分析异常堆栈跟踪
│
├─ 程序变慢（性能问题）
│  ├─ 特定操作慢 → 分析该操作的代码路径
│  ├─ 全局变慢 → 检查系统资源（CPU、内存、I/O）
│  ├─ 间歇性变慢 → 查看日志中的GC或其他定时任务
│  └─ 运行时间越来越长 → 检查是否内存泄漏或资源积累
│
├─ 数据不一致
│  ├─ 并发问题 → 添加同步或事务
│  ├─ 缓存问题 → 检查缓存更新逻辑
│  ├─ 数据库问题 → 检查约束和触发器
│  └─ 应用逻辑错误 → 仔细审视业务逻辑
│
├─ 间歇性错误
│  ├─ 并发相关 → 添加日志和同步
│  ├─ 时序相关 → 检查异步操作的顺序
│  ├─ 资源相关 → 监控资源可用性
│  └─ 环境相关 → 检查网络、DNS、时间同步
│
└─ 无法重现的问题
   ├─ 并发问题 → 压力测试或添加更多日志
   ├─ 环境特定 → 在相同环境中复现
   ├─ 时序问题 → 检查定时任务和异步操作
   └─ 罕见条件 → 分析边界情况
```

## ⚡ 快速诊断技巧

### 技巧1: 三行快速诊断

如果只有3句话空间，说：
1. **现象**: 什么情况下出现问题
2. **日志**: 错误信息是什么
3. **尝试**: 我已经尝试过什么

### 技巧2: 添加诊断日志

```
关键位置添加：
- 函数入口: log("enter getUser with id=" + id)
- 关键决策: log("condition A=" + conditionA + " B=" + conditionB)
- 异常处理: log("exception", error, "context=" + context)
- 出口: log("return " + result)
```

### 技巧3: 二分法快速定位

```
如果问题在大块代码中：
1. 注释掉一半
2. 问题消失→问题在这一半
3. 问题存在→问题在另一半
4. 重复直到精确定位
```

### 技巧4: 最小化复现

```
创建最小化示例：
- 删除业务逻辑，只保留导致问题的部分
- 使用简单数据替代真实数据
- 独立运行，不依赖其他系统
- 同事可以快速复现
```

## 📋 不同场景的诊断Prompt模板

### 当你有错误堆栈时

```
我得到了这个错误：
[粘贴完整堆栈跟踪]

代码：
[相关代码]

问题：
1. 错误的根本原因是什么？
2. 怎样快速修复？
3. 如何预防？
```

### 当你没有错误信息但有日志时

```
根据这些日志，这个系统出了什么问题？
[粘贴日志片段]

时间线：
1. [时间] 用户执行了[操作]
2. [时间] 系统返回[结果]
3. [时间] [问题出现]

请分析：
1. 从日志看问题发生的顺序？
2. 可能的原因是什么？
3. 应该添加什么日志来进一步诊断？
```

### 当问题是性能问题时

```
我的[系统/功能]变慢了。

当前状态：
- 响应时间：[X秒]
- 目标：[Y秒]
- 用户数：[N]

观察：
- [观察1]
- [观察2]
- [观察3]

已排除：
- [已检查的方面]

请帮我诊断：
1. 最可能的原因是什么？
2. 怎样进一步确认？
3. 需要什么工具或指标？
```

### 当问题是并发相关时

```
我的系统在高并发下出现问题：
[问题现象]

核心操作代码：
[代码]

问题：
1. 这是竞态条件吗？怎样触发？
2. 可能导致什么后果？
3. 怎样修复？
4. 怎样验证修复？
```

## 🎓 诊断Best Practice

### ✅ 应该做

- ✅ 提供完整的环境信息（版本、配置）
- ✅ 附上最小化的复现用例
- ✅ 说明问题的频率和规律
- ✅ 展示已经尝试过的诊断步骤
- ✅ 提供相关的日志和指标
- ✅ 明确说明预期行为 vs 实际行为

### ❌ 不应该做

- ❌ 只说"不工作了"或"出错了"
- ❌ 贴整个项目源码，应该贴最小化案例
- ❌ 让AI盲目猜测，应该提供诊断数据
- ❌ 忽视环境差异，应该明确环境
- ❌ 隐瞒已有错误信息，应该完整提供
- ❌ 期望一次就完美解决，应该迭代诊断

## 🚀 常见Bug类型的快速诊断

| 症状 | 快速诊断 | 验证方法 |
|------|---------|--------|
| **程序崩溃+堆栈跟踪** | 找堆栈中的业务代码行 | 检查该行的输入和状态 |
| **性能突然下降** | 检查是否有大查询或GC | 看日志或监控数据 |
| **内存不断增长** | 很可能是内存泄漏 | 导出堆快照分析 |
| **特定条件下出错** | 隔离这个条件 | 调试该条件的代码路径 |
| **间歇性问题** | 可能是并发或时序 | 压力测试或加日志 |
| **升级后出现问题** | 检查兼容性和配置 | 对比之前的版本 |
| **新功能慢** | 检查新增代码的性能 | Profiling找瓶颈 |
