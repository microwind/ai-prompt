# T01: Requirement Clarification（需求澄清）

> ⭐ 初级 | 最关键的第一步：理清需求、确认范围、澄清边界

## 核心概念

许多项目失败都源于需求不清楚。在开始编码前花时间澄清需求，能够：
- 避免做不必要的工作
- 减少返工和变更
- 提高团队效率
- 设定正确的期望

好的需求澄清应该回答这些问题：
1. **What（做什么）**: 核心功能和目标
2. **Why（为什么）**: 业务驱动和用户痛点
3. **Who（谁用）**: 目标用户和角色
4. **How（怎么用）**: 使用场景和流程
5. **When（何时）**: 优先级和截止期限
6. **How much（花多少）**: 资源、成本、范围约束

## 最佳实践

🎯 **原则1: 先问为什么，再问怎么做**
- ❌ 差: "建立一个用户管理系统"
- ✅ 好: "我们有1000个外部合作伙伴需要管理，目前用Excel维护很低效，需要一个系统来简化权限和数据维护"

🎯 **原则2: 列举具体的场景和例子**
- ❌ 差: "用户应该能够管理账户"
- ✅ 好: "用户可以：1)修改昵称 2)更换头像 3)修改密码 4)启用两步验证 5)导出数据"

🎯 **原则3: 明确范围和约束**
- ❌ 差: "需要一个快速的系统"
- ✅ 好: "目标响应时间<200ms，支持1000并发用户，初期需支持Web版，移动版后续"

🎯 **原则4: 确认成功标准**
- ❌ 差: "做好了就上线"
- ✅ 好: "验收标准：1)所有核心场景流程通 2)90%单元测试通过 3)无P1bug 4)性能达标"

## ✅ 优质Prompt示例

### 示例1: 模糊需求的澄清

**Prompt:**
```
我们的产品经理给了一个模糊的需求，请帮我系统地澄清它。

原始需求：
"构建一个内容管理系统来管理公司的博客和资源库"

请帮我澄清以下方面：
1. 用户类型：谁会使用这个系统？（编辑、审核、读者等）
2. 核心功能：必须有哪些功能？哪些是nice-to-have？
3. 内容类型：支持什么样的内容？（文本、图片、视频、PDF等）
4. 工作流：内容从创建到发布的流程是什么？
5. 访问控制：不同用户有什么权限？
6. 规模：预期内容数量？每天多少访问？
7. 集成：需要和其他系统集成吗？
8. 成功标准：什么才算项目成功？

基于你的分析，列出5个我应该继续追问的问题。
```

**预期输出:**
- 清晰的用户画像
- 优先级明确的功能列表
- 具体的范围和约束
- 建议的澄清问题

**适用场景:** 启动新项目、需求描述不清、跨部门协作

### 示例2: 用户故事补充

**Prompt:**
```
这是我们的产品需求，请补充成完整的用户故事。

需求原文：
"系统应该支持用户搜索"

请生成用户故事，格式为：
作为 [用户角色]，
我想 [用户目标]，
以便 [商业价值/好处]

对于"搜索"功能，请生成3-5个不同角色的用户故事，包括：
- 每个故事的验收标准（至少3个）
- 可能的例外情况或边界条件
- 优先级建议

示例：
作为 博客读者，
我想 用关键词搜索感兴趣的文章，
以便 快速找到我需要的内容

验收标准：
1. 可以按标题和内容搜索
2. 搜索结果按相关性排序
3. 搜索可以过滤日期范围
```

**预期输出:**
- 格式规范的用户故事
- 明确的验收标准
- 优先级指导

### 示例3: 场景边界确认

**Prompt:**
```
我们在开发一个支付功能，请帮我确认所有边界情况。

核心需求：用户可以用信用卡支付订单

请列出：
1. 应该支持的场景（正常流程）
   - 支持哪些卡类型？
   - 支持哪些货币？
   - 最小/最大金额限制？

2. 应该处理的异常情况
   - 卡被拒绝
   - 超时
   - 网络错误
   - 用户取消
   - 重复支付

3. 不在范围内的场景（明确说NO）
   - 是否支持预授权？
   - 是否支持分期付款？
   - 是否支持货币兑换？
   - 是否支持退款？

4. 依赖的外部系统
   - 支付网关（哪个提供商？）
   - 数据库
   - 日志系统

针对每个场景，说明预期的用户体验和系统行为。
```

**预期输出:**
- 清晰的范围边界
- 各类场景的处理方式
- 依赖和约束列表

### 示例4: 非功能需求列举

**Prompt:**
```
我们需要明确这个项目的非功能需求。

功能需求：用户认证和授权系统

请帮我定义：

1. 性能要求
   - 登录响应时间：[多少ms以内？]
   - 并发用户：[支持多少？]
   - 吞吐量：[每秒多少请求？]

2. 可用性
   - 目标SLA：[99.9%还是99.99%？]
   - 维护窗口：[允许多久不可用？]
   - 灾难恢复RTO/RPO：[多久恢复？数据丢失容限？]

3. 安全性
   - 密码策略：[复杂度要求？]
   - 会话超时：[多少分钟？]
   - 加密传输：[必须HTTPS吗？]
   - 审计日志：[记录哪些操作？]

4. 扩展性
   - 预期用户增长：[1年增长到多少？]
   - 数据增长：[多少GB/年？]
   - 峰值流量倍数：[是平均的几倍？]

5. 可维护性
   - 代码覆盖率目标：[多少%？]
   - 文档要求：[API文档？架构文档？]
   - 监控和告警：[需要什么指标？]

6. 合规性和成本
   - 数据隐私法规：[GDPR/CCPA/等？]
   - 预算限制：[基础设施成本上限？]
   - 团队大小：[多少人维护？]

对于每个要求，给出具体的数字或标准。
```

**预期输出:**
- 量化的性能指标
- 明确的SLA目标
- 安全和合规要求
- 成本预算

### 示例5: 可行性快速评估

**Prompt:**
```
在我们投入开发前，请帮我快速评估这个功能的可行性。

需求：在我们的电商平台上添加"直播购物"功能
预期投入：2个月，5个人

可行性评估维度：
1. 技术可行性
   - 我们的技术栈能支持吗？
   - 需要学习新技术吗？
   - 有现成的解决方案吗？

2. 时间可行性
   - 2个月实现所有功能现实吗？
   - 最小化可行产品(MVP)需要多长时间？
   - 有哪些可以延后的功能？

3. 资源可行性
   - 5个人够吗？需要什么角色？
   - 是否需要第三方支持？
   - 基础设施成本是多少？

4. 市场可行性
   - 竞争对手已经做了吗？
   - 我们有竞争优势吗？
   - 用户真的需要吗？

5. 风险评估
   - 最大的风险是什么？
   - 如何降低风险？
   - 有备选方案吗？

基于以上分析，给出：
- 整体可行性评分（1-10）
- 建议的实施方案
- 关键成功因素
- 建议延后的功能
```

**预期输出:**
- 结构化的可行性评估
- 风险识别和缓解方案
- MVP方案建议
- 实施路线图

### 示例6: 风险识别

**Prompt:**
```
在这个项目正式启动前，请帮我识别所有可能的风险。

项目简介：
- 名称：[项目名]
- 目标：[目标描述]
- 时间：[时间框架]
- 人员：[团队组成]
- 预算：[预算信息]

请从以下维度识别风险：

1. 技术风险
   - 使用的技术成熟吗？
   - 团队有经验吗？
   - 有技术债吗？
   - 第三方依赖可靠吗？

2. 进度风险
   - 时间估计有信心吗？
   - 有不确定的部分吗？
   - 依赖其他团队吗？

3. 资源风险
   - 人员是否充足？
   - 是否有关键人员离职风险？
   - 基础设施是否充分？

4. 业务风险
   - 需求会变化吗？
   - 市场情况会改变吗？
   - 竞争对手会怎么做？

5. 外部风险
   - 供应商或合作方可靠吗？
   - 法规会改变吗？
   - 安全隐患？

对于每个识别的风险，给出：
- 概率（高/中/低）
- 影响（高/中/低）
- 风险等级（根据概率×影响）
- 缓解方案

最后，列出TOP 5最关键的风险和应对计划。
```

**预期输出:**
- 风险列表及分类
- 风险等级矩阵
- 缓解计划
- 应急预案

## ❌ 常见误区

### 误区1: 急于动手，跳过澄清

❌ 不要：
```
"需求我看过了，没问题，开始开发吧"
[2周后] "咦，需求和理解不一样..."
```

✅ 应该：
```
花1-2天澄清需求，确保团队的理解一致
用书面形式记录澄清的结果
让相关方确认和签字
```

### 误区2: 澄清不充分

❌ 不要：
```
"构建一个用户系统"
就开始开发，没有明确：
- 用户数量
- 性能要求
- 安全要求
- 扩展性需求
```

✅ 应该：
```
明确的量化指标：
- 支持100万用户
- 登录<100ms
- 支持10,000 QPS
- 99.99% SLA
```

### 误区3: 遗漏异常情况

❌ 不要：
```
只想快乐路径，不考虑：
- 网络故障
- 数据库连接失败
- 用户误操作
- 超大数据
```

✅ 应该：
```
列出所有可能的异常情况
定义每个异常的处理方式
明确系统行为
包括回滚和恢复机制
```

## 高级技巧

### 技巧1: 使用需求模板

```
功能需求
--------
功能名称：[名称]
目的：[为什么需要]
用户：[谁用]

场景：
1. [场景1及流程]
2. [场景2及流程]
...

数据：
- 输入：[输入数据]
- 输出：[输出数据]
- 存储：[需要保存什么]

约束：
- 性能：[性能要求]
- 安全：[安全要求]
- 兼容性：[兼容要求]

范围外：
- [不需要实现的功能]
```

### 技巧2: 用原型或mockup澄清

```
有些东西用语言很难说清，建议：
1. 画个草图或用设计工具做mockup
2. 让相关方确认和调整
3. 这样可以节省大量沟通时间

Prompt模板：
"基于这些需求，请生成一个[功能]的UI mockup描述，
包括主要的页面和交互流程"
```

### 技巧3: 优先级矩阵

```
对于有多个功能的需求，用优先级矩阵梳理：

Prompt:
"这个项目有很多功能需求，请帮我用优先级矩阵分类。

功能列表：
1. [功能1]
2. [功能2]
...

用以下维度评估：
- 对用户价值：高/中/低
- 实现难度：高/中/低

基于这两个维度，给出：
- P0（必须有）
- P1（应该有）
- P2（不必急）

并建议分阶段实现的计划"
```

## 📌 需求澄清检查清单

启动项目前，确认：
- [ ] 业务目标清晰（Why）
- [ ] 核心功能明确（What）
- [ ] 用户群体确定（Who）
- [ ] 使用场景定义（How）
- [ ] 成功标准明确（完成后如何验证）
- [ ] 非功能需求量化（性能、安全、规模等）
- [ ] 范围边界清楚（在范围内和范围外都清楚）
- [ ] 依赖关系识别（需要其他团队或系统配合的地方）
- [ ] 风险已识别（主要风险和应对方案）
- [ ] 所有相关方确认（签字或邮件确认）
